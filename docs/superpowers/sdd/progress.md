## Operational notes
- **Set component properties via `Unity_RunCommand` (direct C#), NOT `ManageGameObject` component_properties.** On `create`, component_properties were silently ignored (Canvas came out WorldSpace/800x600); typed enums don't map. RunCommand with real enum values is deterministic. Applies to all prefab-building tasks (RectTransform/Image/TMP props).
- **UI CAPTURE (portrait, reliable): use the self-contained RunCommand in `docs/superpowers/sdd/capture-helper.txt`** → writes `Temp/uicap.png` (540x960), then Read it. Deliverable canvas stays Screen Space - Overlay; helper flips to camera+RT transiently and restores. Temp/ is gitignored.
- **Capture pitfalls (DO NOT REPEAT):** (a) `Unity_Camera_Capture` no-id = scene-view = shows Overlay canvas at Game-View landscape aspect w/ skybox — NOT usable for portrait UI. (b) `Unity_Camera_Capture(camId)` FAILS if that camera has a targetTexture ("Failed to render scene preview"). (c) ANY RunCommand that touches the Game View (GetWindow(GameView) / selectedSizeIndex / Repaint) → breaks the MCP harness ("UNEXPECTED_ERROR: Object reference not set"). (d) `AssetDatabase.DeleteAsset` on a folder → "User interactions are not supported" — delete assets via filesystem (Bash rm) instead.
- **In RunCommand, the injected namespace makes `Image` (and likely other UI types) AMBIGUOUS with Unity.AI.* — alias: `using UIImage = UnityEngine.UI.Image;`. Also avoid `typeof(Editor)`/`typeof(Image)` for assembly; use `typeof(EditorWindow).Assembly`. Don't pass non-Unity objects (Type/ConstructorInfo) to result.Log — it can NRE the harness.**
- Build prefabs via RunCommand C# builders (create GameObjects/components, set props, PrefabUtility.SaveAsPrefabAsset) — token-light + deterministic vs many ManageGameObject calls.
- **BEFORE running EditMode `run-tests`, SAVE THE SCENE (EditorSceneManager.SaveOpenScenes via RunCommand).** A dirty scene makes the test runner's domain reload pop a modal "Scene(s) Have Been Modified" save dialog that BLOCKS Unity → run-tests times out (30s/90s) and needs the user to click a button. Play-mode + capture leaves the scene dirty. (Happened at Task 13; Omar dismissed the modal.)
- Play-mode transition test pattern (worked): ManageEditor Play → ReadConsole errors → capture (helper) initial → RunCommand `ScreenManager.Instance.ShowOverlay(...)` → capture end-state (tween 0.35s < command latency) → ManageEditor Stop.
- **Unity bridge CLI is NOT on PATH.** Invoke as a module WITH utf-8 (Windows console chokes on ✓):
  `PYTHONIOENCODING=utf-8 PYTHONUTF8=1 python -m claude_unity_bridge.cli <get-status|refresh|run-tests|compile|play|pause> [args]`
  Test-run flow after writing .cs: `refresh` → poll `get-status` until "Compilation: ✓ Ready" → `run-tests --mode EditMode --filter <Name>` (run-tests errors if run mid-compile).
